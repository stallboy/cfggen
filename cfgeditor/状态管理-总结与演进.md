## 一、本质问题：React 应用里"状态"到底有几种？

新人常把"状态管理"等同于"选一个 Redux/Zustand"。这是窄化理解。一个真实应用的**状态来自好几个性质截然不同的源**，混用同一个工具是绝大多数 bug 的根。

按"谁拥有它、是否异步、生命周期、是否需要响应式"分类：

| 状态源 | 谁拥有 | 异步? | 生命周期 | 典型例子 |
|---|---|:---:|---|---|
| **服务端状态** | 后端 | ✅ | 与请求/缓存绑定 | 一条 record、schema、引用列表 |
| **URL 状态** | 浏览器地址栏 | ❌ | 跨刷新/跨分享 | 当前 table/id、是否编辑模式 |
| **客户端 app 状态** | app 全局 | ❌ | app 会话内 | 主题、布局参数、当前打开的面板 |
| **会话/表单状态** | 某次交互 | ❌ | 进入→离开该交互 | 一个 record 的编辑草稿 |
| **局部 UI 状态** | 单个组件 | ❌ | 组件挂载期 | 输入框值、下拉开关 |
| **持久化状态** | 磁盘/存储 | ✅(写) | 跨重启 | 上次的设置、导航历史 |

**关键判据**：服务端状态首选专门的 server-state 库（React Query/SWR），**不要**塞进全局 store；URL 能承载的状态优先放 URL（可分享、可前进后退）；只有"客户端独有的、需要跨组件响应式广播的"才进全局 store；"这次交互独有、性能敏感的"用专门的会话 store。

> 一句话：**先分清状态属于哪一类，再选工具。选错源 = 选错工具 = 后面全是补丁。** cfgeditor 的 `startEditingObject` 反模式（第八节）就是典型的"把会话状态放进了全局单例"。

## 二、主流状态方案谱系（一句话定位）

| 方案 | 本质 | 适用 | 不适用 |
|---|---|---|---|
| `useState`/`useReducer` | 组件内响应式 | 局部 UI、简单派生 | 跨组件共享 |
| Context | 跨组件"穿洞"传值 | 低频变更的依赖注入（主题/路由） | 高频变更（全树重渲） |
| Redux | 单一不可变 store + action | 大型、可时间旅行/审计的应用 | 小项目过重 |
| Zustand | 极简外部 store + hook | 通用客户端状态 | — |
| **resso** | **每 key 一个外部 store 的工厂** | **轻量客户端状态（本项目选型）** | 服务端状态 |
| Jotai/Recoil | 原子化、依赖追踪 | 细粒度派生多的场景 | 概念成本 |
| **React Query/SWR** | **服务端状态缓存 + 同步** | **任何后端数据（本项目选型）** | 纯客户端 UI 状态 |
| `useSyncExternalStore` | **把"外部可变状态"接进 React 的官方接口** | 订阅式外部 store、命令式引擎 | 需自己管订阅/snapshot |

注意最后一行：`useSyncExternalStore` 不是"又一个状态库"，而是**所有外部 store（包括 resso、Zustand、React Query 底层）接入 React 的共同底座**。看懂它就看懂了 cfgeditor 的整个响应式骨架——这是第四节的主题。

## 三、cfgeditor 当前状态全景

| 状态源 | 工具/文件 | 作用域 | 响应式机制 | 代表字段 |
|---|---|---|---|---|
| 服务端数据 | React Query（`queryClient.ts`） | app | query 缓存 + hook 订阅 | `schema`/`record`/`refs`/`prompt`/`layout` |

| URL/路由 | react-router（`useLocationData`） | URL | `useLocation` | `curPage`/`curTableId`/`curId`/`edit` |
| 持久化 | `store/storage.ts` | localStorage + Tauri YAML | 非响应式（`setPref` 同步） | resso 的可持久化子集 |
| 局部 UI | `useState`/`useRef` | 组件 | React 内置 | `folds`/`inputValue` |

注意最后两行和 EditingSession 的关系：**EditingSession 持有真正的编辑对象，resso 只持有一个"当前在编辑谁、是否脏"的指针**。这是后面要反复讲的设计要点。

## 四、一把钥匙：`useSyncExternalStore`

cfgeditor 的两个核心状态机制（resso、EditingSession）**底层是同一个 API**。理解它，整个 app 的响应式就通了。

### 4.1 它解决什么问题

React 的 render 必须是纯函数（不读不写外部世界）。但"命令式外部状态"（WebSocket、编辑器实例、全局 store、模块单例）天然可变。`useSyncExternalStore` 是 React 官方提供的**合法逃生通道**——把外部可变状态以响应式方式接进 render：

```ts
const value = useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot?);
//  subscribe: (cb) => 取消订阅   —— 外部变了时调 cb 通知 React
//  getSnapshot: () => 快照        —— React 用它读当前值，并靠"引用是否变了"判断要不要重渲
//  getServerSnapshot: SSR 时的快照 —— SPA 可不传，或传同一个 getSnapshot
```

### 4.2 两条铁律（违反即 bug）

1. **`getSnapshot` 必须返回"引用稳定"的值**：同一份状态没变时，两次调用必须返回 `===` 相等的值。否则 React 以为变了 → 无限重渲。**基本类型（number/string）天然满足**；返回对象/数组则必须缓存。
2. **`subscribe`/`getSnapshot` 引用要稳定**：每次 render 传给 `useSyncExternalStore` 的这两个函数应是同一个引用（否则每次都重新订阅，带来无谓的订阅抖动）。

> 顺带一提：resso 用的是 `use-sync-external-store/shim`（`resso.ts:2`）而非 `react` 直接导出版——shim 是给 React <18 的 polyfill，React 18+ 有内置版，行为一致；resso 作为通用库选 shim 兼容旧版，本项目用 React 19 不受影响。

### 4.3 resso = 每个 key 一个 useSyncExternalStore（`store/resso.ts`）

resso 是个 ~150 行的 vendored 库。它的精髓是：**store 的每个 key 各自是一个独立的外部 store**。

```ts
const store = resso({ server: 'x', maxImpl: 10 });

// 组件 A
const { server } = useMyStore();   // render 时访问 store.server → 订阅 server 这一个 key
// 组件 B
const { maxImpl } = useMyStore();  // 订阅 maxImpl

setServer('y');   // 只 triggerUpdate(server) → 只 A 重渲
setMaxImpl(20);   // 只 triggerUpdate(maxImpl) → 只 B 重渲
```

实现上（`resso.ts:87-105`）是个 Proxy：读 `store.foo` 时调 `useSyncExternalStore(state.foo.subscribe, state.foo.getSnapshot, state.foo.getSnapshot)`；写 `store.foo = v` 时更新值并 `triggerUpdate`。`getSnapshot` 返回基本类型或稳定引用 → 天然满足铁律 1。

> **这就是为什么 cfgeditor 敢把几十个配置字段塞进一个大 `StoreState`**：resso 的订阅是 per-key 的，改 `maxImpl` 不会让只读 `server` 的组件重渲。不存在"大 store = 全树重渲"的代价。

### 4.4 EditingSession = 手写的一个 useSyncExternalStore store（`services/editingSession.ts`）

```ts
class EditingSession {
    private structureVersion = 0;                    // 快照：基本类型，天然稳定
    private listeners = new Set<() => void>();
    subscribe = (l) => { this.listeners.add(l); return () => this.listeners.delete(l); };
    getStructureVersion = () => this.structureVersion;  // 引用稳定的箭头函数属性
    // ...
}
```

`Record.tsx:89`：
```ts
const structureVersion = useSyncExternalStore(session.subscribe, session.getStructureVersion);
```

和 resso 完全同构——只不过 resso 自动给每个 key 生成这套，EditingSession 是为"编辑会话"这一个特殊场景手写的。**两者遵守同一对铁律**：`subscribe`/`getStructureVersion` 是实例箭头函数属性（绑定 `this` + 引用稳定，满足铁律 2）；`getStructureVersion` 返回 number（满足铁律 1）。

> ⚠️ 维护提示：`subscribe`/`getStructureVersion` 必须保持**实例箭头函数属性**写法。若改成 prototype 普通方法，`this` 会丢、引用也不稳定，`useSyncExternalStore` 立刻崩。

## 五、深入 resso store（`store/store.ts`）

### 5.1 StoreState 的三类字段

一个 `StoreState`（`store.ts:32`）混装了三类内容：

| 类别 | 字段 | 是否持久化 |
|---|---|---|
| **配置** | `server`/`aiConf`/`themeConfig`/`nodeShow`/`tauriConf`/各种 depth/maxNode | `cfgeditor.yml`（共享） |
| **UI 偏好** | `isEditMode`/`dragPanel`/`pageConf`/`imageSizeScale`/`query` | `cfgeditor.yml` 或 `cfgeditorSelf.yml` |
| **运行时指针** | `editingCurTable`/`editingCurId`/`editingIsEdited`/`history`/`resMap`/`resourceDir` | **不持久化**（`notSaveKeySet`） |

持久化三分（`store.ts:157-161`）：
- **共享** `cfgeditor.yml`：`getPrefKeySet` = 全部键 − 个人键 − 不持久化键。
- **个人** `cfgeditorSelf.yml`：`prefSelfKeySet` = `{curPage, curTableId, curId, query, isEditMode, imageSizeScale, dragPanel, aiConf}`（`aiConf` 含 apiKey，单独放个人文件）。
- **不持久化** `notSaveKeySet` = `{history, resMap, resourceDir, editingCurTable, editingCurId, editingIsEdited}`（每次启动重算/重取）。

### 5.2 setter 的标准三段式

几乎每个 setter 都是同一模式（例 `setMaxImpl`，`store.ts:280`）：
```ts
export function setMaxImpl(value: number | null) {
    if (value) {
        store.maxImpl = value;          // ① 改 resso（触发订阅者重渲）
        setPref('maxImpl', value.toString());  // ② 持久化（localStorage + debounce 写 YAML）
        clearLayoutCache();             // ③ 清 React Query 的 layout 缓存（拓扑参数变了，布局要重算）
    }
}
```

这三步体现了一个跨源协作：**resso（UI 配置）变了 → storage（持久化）写盘 → React Query（派生数据）缓存失效重取**。三个状态源在一次操作里联动。

### 5.3 持久化写入：防抖 + 串行化（`store/storage.ts`）

- **双存储**：web 用 localStorage；Tauri 下启动时读 YAML → 灌入 localStorage（`readPrefAsyncOnce`），`setPref` 写 localStorage + 防抖写 YAML。
- **防抖 300ms**（`storage.ts:119`）：`navTo` 一次连发 3 个 `setPref`，防抖合并成一次 YAML 全量重写。
- **串行化**（`storage.ts:110`）：`writeChain` 串成 Promise 链，避免并发写同一文件损坏/丢字段。

### 5.4 `useMyStore` vs `getMyStore`

```ts
export function useMyStore() { return store; }   // 在组件里调，建立 per-key 订阅
export function getMyStore() { return store; }   // 在非组件（事件回调/util）里直接读，不订阅
```

同一个 store 代理，区别只在于调用场景：组件里要响应式就用 `useMyStore`；纯工具函数/事件回调里偶尔读一下当前值用 `getMyStore`（不引发订阅）。注意"不引发订阅"是因为 `getMyStore` 虽返回同一个 Proxy，但**不在组件 render 期调用**——resso 的订阅发生在 render 期属性访问时，非 render 期访问只读值不订阅。

## 六、深入 EditingSession（`services/editingSession.ts`）

### 6.1 为什么编辑对象不进 resso / useState？

这是整个编辑管线最关键的设计决策。表面看 `editingObject` 是"客户端状态"，进 resso 天经地义。但 cfgeditor 偏要为它单造一个 `EditingSession` 类，原因是**两条性能契约**：

- **契约 1（值类零重渲）**：一个 record 可能有几十个表单字段。用户连续键入时，几十个 `EntityForm` **不能重渲**，否则卡顿。键入时输入框 UI 由 antd Form 自己管（`initialValue` + `setFieldValue`），`editingObject` 被就地改但 React 看不见 → Record 不重渲 → 表单不重渲 → 输入流畅。
- **契约 2（共享引用）**：值类编辑就地改 `editingObject`，所有 entity 闭包持有它的子对象引用（`edit.editObj = obj`），改完自动见最新值，**不必重算整棵实体图**；提交时 `submit()` 读到全量最新。

resso 是"不可变 + 赋新值触发订阅"语义。若 `editingObject` 进 resso：每次键入 `store.editingObject = newObj` → 引用变 → 所有依赖重算 → 几十表单重渲，且 antd Form 的 `useEffect([field.value])` 会 `setFieldValue` 与用户正在输入的值打架（光标跳、值被覆盖）。**与契约 1 直接冲突。**

EditingSession 的解法：**就地变异（不换引用）+ 结构版本号（选择性 bump）**：
```ts
// 值类（键入 primitive / 改 note）：就地改，不 bump
updateFormValues(...) { /* obj[key] = conv(v) */ this.notifyEditingState(); }  // 只刷脏标记，不 emit
// 结构类（增删/swap/fold/impl/paste/replace）：就地改 + bump
addArrayItem(...) { obj.push(item); this.structureChange(pos); }  // bump + emit → Record 重渲
```

`Record.tsx` 只订阅 `structureVersion`（number）。值类编辑不 bump → Record 不重渲 → 表单不重渲（契约 1 保住）；结构类编辑 bump → Record 重渲 → entityMap 重算时读最新 `editingObject`（共享引用，契约 2 保住）。

> **教学点**：同一个 app 里，resso 服务"低频、可不可变化"的配置状态；EditingSession 服务"高频、必须就地变、选择性通知"的会话状态。**选对工具**比"统一用一个 store"重要得多。

### 6.2 EditingSession 与 resso 的桥：`editingIsEdited`

EditingSession 自持 `getIsEdited()`，但 HeaderBar（显示"未保存"脏标记的组件）在**另一个路由子树**，拿不到这个会话实例。解法是搭一座桥：

```ts
// editingSession.ts:252 —— 值类/结构类编辑都调
notifyEditingState() { setEditingState(this.table, this.id, this.getIsEdited()); }
// store.ts:507
setEditingState(...) { store.editingCurTable = ...; store.editingCurId = ...; store.editingIsEdited = ...; }
```

于是脏标记通过 **resso 的 `editingIsEdited` 广播给 HeaderBar**，而真正的编辑对象留在 EditingSession 里。这是"会话状态（精确）+ app 指针（广播）"的分工。

注意存在**两个 isEdited 通道**（项目内既有设计决策，有意保留）：
1. resso `editingIsEdited` → HeaderBar 实时脏标记；
2. `editingObjectRes.isEdited`（`editingSession.ts:86`）→ `useEntityToGraph` 决定 layout 的 `staleTime`（脏了就每帧重取布局，干净就 5min 缓存）。

通道 2 在纯值类编辑期间不刷新（entityMap 不重算 → `getEditingObjectRes` 不被调），是有意 quirk：值类不改拓扑，布局不必重算，layout 继续走 5min 缓存是**正确**的。勿当 bug 修。

### 6.3 输入净化：`$refs` 剥离

EditingSession 的 `prepareEditingObject`（构造期 / `maybeReset`）和 `replaceEditingObject`（Chat/AddJson/funcClear 写入）都会调 `deleteRefsInPlace` 剥离后端附加的 `$refs`（`FieldRef[]`，"哪些记录引用了本记录"的展示元数据）。`$refs` 是运行时引用关系、非可编辑数据，不剥离会污染提交载荷并让 `getIsEdited` 误判 dirty（基准 `originalEditingObject` 已净化）。注意它不是 JSON Schema 的 `$ref`（引用指针）——命名上的相似是历史包袱。（`deleteRefsInPlace` 原名 `delete$refInPlace`，已于近期改名 + 给 `replaceEditingObject` 补上净化。）

## 七、数据流实例：一次编辑的全链路

把前面所有概念串起来。追踪用户在 record 编辑模式里**改一个 primitive 字段**：

```
用户键入
  → antd Form 受控更新输入框（Form 自管值，React 不参与）
  → antd 触发 onFinish/onValuesChange
  → entity.edit.editOnUpdateValues(values)
  → session.updateFormValues(schema, values, fieldChain)        【EditingSession】
      → 定位子对象 + 类型转换 + 就地改 editingObject[key]
      → notifyEditingState()
          → setEditingState(table, id, isEdited)                【resso store】
              → store.editingIsEdited 变
              → resso triggerUpdate(editingIsEdited)
              → HeaderBar（订阅者）重渲 → 显示 "unsaved" 脏标记
      → 【不 bump structureVersion、不 emit】
  → Record 的 useSyncExternalStore 未被通知 → Record 不重渲
  → 几十个 EntityForm 不重渲 → 键入流畅（契约 1）
```

对比**结构类编辑**（加一个数组项）：

```
右键"添加" → session.addArrayItem(default, chain, pos)           【EditingSession】
  → obj.push(item)  就地改
  → structureChange(pos): fitView=FitId, fitViewToIdPosition=pos
  → bumpStructure():
      → structureVersion++                                       【快照变】
      → onStructureChange() → queryClient.removeQueries(['layout', pathname, 'e'])  【React Query 缓存失效】
      → notifyEditingState()                                     【resso 脏标记】
      → emit() → listeners 通知 Record
  → Record 的 useSyncExternalStore 被通知 → Record 重渲
  → useMemo 重算（structureVersion 依赖变）:
      → new RecordEditEntityCreator(entityMap, ..., session, editingObject).createThis()
      → entityMap 重建（闭包持有最新子对象引用，契约 2）
      → editingObjectRes = session.getEditingObjectRes()  (fitView=FitId)
  → useEntityToGraph 收到新 entityMap + editingObjectRes
      → layout query 因缓存被清 → 重取 → FitId 定位到新节点
```

这两条链路串起了 **EditingSession × resso × React Query × antd Form × react-router(URL)** 五个状态源。看懂它，就看懂了 cfgeditor 的运行时。

## 八、反模式案例：`startEditingObject`（已根治）

这一节作为"状态建模出错会怎样"的教学案例——它是 cfgeditor 真实踩过、并已根治的坑（重构落在 commit `f6d27484` / `9f542399` / `3ab0527a`）。

### 8.1 病根：把会话状态放进模块级全局单例，并在 render 期变异它

旧版把编辑会话状态建模成**模块级全局单例 `editState`**，并在 `Record.tsx` 的 `useMemo`（= render 阶段）里调用 `startEditingObject()` **直接变异**它，紧接着同一段 `useMemo` 里 `createThis()` 又读回这个刚被写过的全局构建实体图。这同时违反了第一节的多条判据：

1. **选错源**：会话状态（应每会话独立）被放进了 app 全局单例 → 跨路由（Chat/AddJson 共用它）隐式耦合、潜在分屏多 Record 互踩。
2. **render 期副作用**：render 必须纯，它却写外部世界（全局可变状态）→ 丢失 React Compiler 等价性保证、StrictMode 下靠"幂等覆盖"侥幸不炸。
3. **无订阅层**：全局不是响应式的，靠 `update()` 引用代理这个旁路"假装"响应式 → 声明依赖 ≠ 真实依赖。

整套链路靠一条承重不变量勉强工作："一次编辑会话内 `startEditingObject` 永远走内容未变的早退分支 → `editingObject` 在两次 `update()` 间永不被替换"。这条不变量没有任何类型或工具把关，是脆弱的巧合。

### 8.2 为什么"挪进 useEffect"和"上 reducer"都不是正解

直觉上常见的两种修法，在本场景都不可行：

- **方案 A（挪进 `useEffect`）❌**：同一段 `useMemo` 里的 `createThis()` 仍读全局。effect 在 commit **之后**才跑，而 `useMemo` 在 render **期间**就已经用旧/空全局构建了实体图 → 首屏错图，且 effect 跑完没有任何依赖变化能触发重算，除非再引入一帧延迟的 forceUpdate。只是把脏挪了位置，不解决根因。
- **方案 B（reducer state，朴素形态）❌**：reducer 的规矩是**不可变更新**，每次产出新引用。这跟 §6.1 的两条性能契约根本冲突——`editingObject` 每次编辑都是新引用 → `useMemo` 必然重算 → 几十表单重渲 → antd Form 与输入打架。朴素 reducer 不可行。
- **方案 B'（reducer 修正形态：分层 state + 结构签名）⚠️**：让 `entityMap` 的 `useMemo` 只依赖一个"结构签名"而不依赖 `editingObject`，值类 dispatch 不动结构签名 → 不重算。可行但要**重造**方案 C 的能力（就地变异 + 选择性订阅），并接受额外复杂度（深嵌套不可变更新、lint disable 有意漏依赖），性价比不如 C。
- **方案 C（`useSyncExternalStore` + 每会话 store）✅ 正解**：把 `editState` 单例重构成**每会话一个**的、带订阅能力的 store 实例（即第六节的 EditingSession）。它**完整保留两条性能契约**（就地变异 + 选择性订阅 + 共享引用），只移除 render 期副作用，`recordEditEntityCreator` 改动最小。

### 8.3 正解与教训

方案 C 已实施：副作用全部移出 render，落在构造期 / effect / 事件回调；每会话独立实例天然支持分屏状态隔离；`useSyncExternalStore` 让"外部可变状态"合法接入 React。

> 一句话教训：**可变单例不是反模式；render 期变异它 + 裸单例无订阅层被组件直读，才是。** cfgeditor 的痛苦不来自"用了可变单例"，而来自"用得不够彻底"——既没把它包成有订阅的 store，又让它越界在 render 期初始化。业界（Monaco/CodeMirror/Redux/Zustand）大量使用可变单例，但它们**都配一个订阅/响应式接入层**。

## 九、下一步重构优化建议

本节是完整路线图，分"EditingSession 内部健壮性""测试缺口""状态架构""验证清单"四块。优先级标在标题。

> **实施状态（2026-07-11）**：

> - **9.2 ✅** updateFormValues 类型转换（`toInt`/`toFloat` NaN→0、`list<int>` 过滤 undefined）、`$impl` 早退、pasteStruct 深拷贝——端到端用 fixtures 造 Schema，**不 export 私有函数**（测真实路径，设计更干净）。
> - **9.5 ✅** useEntityToGraph layout 缓存策略注释——`'e'` = 编辑态缓存隔离标记、staleTime 脏 0/干净 5min、值类编辑不刷新 isEdited 的 quirk 安全性。

> - 9.1a（holder 分屏）/ 9.3（isEdited 收敛）/ 9.6（store 切片）待定。

### 9.1 [P0] EditingSession 内部健壮性

**(a) holder 单值与分屏 gap**：`getCurrentEditingSession`/`setCurrentEditingSession`（`editingSession.ts:280`）是 app 级模块单例、非响应式。每会话实例已隔离了**状态**（分屏 A/B 的 `editingObject` 不互踩），但"寻址当前会话"仍是单值：分屏同挂两个 Record 时，Chat/AddJson 只能命中最后注册的会话。短期建议把相关文档/注释里"天然支持分屏"的措辞收敛为"状态隔离支持分屏；跨面板写入仍单值"；长期（若要做分屏）holder 改成按面板/tab 寻址（`Map<panelKey, EditingSession>`）。

**(b) `updateFormValues` 的 `$impl` 早退分支**：表单提交的 `values` 若含 `$impl` 且与当前 `$type` 末尾名不一致，整个函数 `return`（防 antd Form 在 impl 切换瞬间残留旧字段值）。这是较强副作用（丢全部普通字段更新），且无测试。建议补单测固化（带 `$impl` 且不一致 → editingObject 不变），注释补一句"整段跳过是为拦截 impl 切换过渡帧的残留字段"。

**(c) `onStructureChange` 同步 `removeQueries` 的选型理由需落注释**：每个结构方法末尾同步调 `queryClient.removeQueries(['layout', pathname, 'e'])`（`Record.tsx:83`）。为什么用 `remove` 而非 `invalidate`：remove 立即删缓存 → 下一帧 layout query 必 miss → 重取，与 `emit()` 同帧，时序确定；invalidate 会主动触发 refetch，与组件重渲可能竞态。且这步**不能挪 effect**——React Query 不因 queryFn 闭包里 nodes 变化重执行，effect 会晚一帧布局错乱。建议在代码现场补注释，防止后人"顺手优化"成 invalidate 或挪 effect 触发一帧错乱（历史上同类坑有过 revert）。

### 9.2 [P1] 测试缺口（兑现"可单测"承诺）

EditingSession 重构的核心收益之一是"编辑链可单测"，目前 `editingSession.test.ts` 覆盖了结构/版本号语义 + fuzz，但**逻辑密度最高的几段还没测**：

- **`updateFormValues` 的类型转换**：`toInt`（`parseInt` NaN→0）、`toFloat`、`getFieldPrimitiveTypeConverter`（int/long/float/list<int> 分发）、数组过滤 undefined。纯函数有判定逻辑却无测试。建议补：合法值转换、非法字符串→0、数组含 undefined 被过滤（顺带固化 9.1b 的 `$impl` 早退）。
- **`pasteStruct` 深拷贝独立性**：粘贴时 `structuredClone(copied)`，若漏 clone 会导致两处共享引用、联动变异。建议补：`structCopy(A)` → 粘到 P1 → 改 P1 → 再粘到 P2，断言 P2 仍是原始副本。

> 已完成项：`deleteRefsInPlace` 改名 + `replaceEditingObject` 补 `$refs` 净化 + 对应单测已落地（见 §6.3）。

### 9.3 [中] `isEdited` 三处判定的收敛评估

当前脏判定散在三处：resso `editingIsEdited`、`EditingSession.getIsEdited()`、`editingObjectRes.isEdited`。它们各有正当用途（广播 / 会话内查询 / layout 缓存策略），不是冗余。但"同一语义三份计算"是可读性负担，且 `getIsEdited` 每次值类编辑跑 O(n) 深比较。建议：先补测试固化三者各自契约（§9.2），再评估能否让 layout 通道复用一个缓存的 dirty 标记而非独立深比较。**不急于合并**——性能契约使它们有意分离。

### 9.4 [中] `historyModel` 纯模型下沉 `domain/`（✅ 已实施）

`History`/`HistoryItem` 是**纯数据结构 + 纯规则**（零依赖、不可变）。按"纯 → domain"的架构判据，归位 `domain/historyModel.ts`。**首要判据是架构一致性**——纯模型不属 `store/` 层（与已下沉的 `folds.ts` 同理）；"它本就已有完整测试"不构成留在 `store/` 的理由。设计标准优先于"是否已可测"。

### 9.5 [中] layout 缓存策略的三方耦合显式化

## 一页速记

- **状态分六类**：服务端(Query)/URL(Router)/客户端app(resso)/会话(EditingSession)/局部(useState)/持久化(storage)。**先分类，再选工具。**
- **一条底座**：resso 和 EditingSession **底层都是 `useSyncExternalStore`**；`getSnapshot` 引用稳定 + `subscribe` 引用稳定 是两条铁律。
- **resso 的精髓**：per-key 订阅——改一个字段只通知订阅它的组件，大 store 无重渲代价。
- **EditingSession 的精髓**：就地变异 + 结构版本号 bump——值类零重渲、结构类重算，保住几十表单的性能。
- **两座桥**：EditingSession 通过 resso `editingIsEdited` 广播脏标记给 HeaderBar；通过 `editingObjectRes.isEdited` 驱动 layout 缓存策略。
- **输入净化**：`$refs`（后端引用元数据，非 JSON Schema `$ref`）在 `prepareEditingObject`/`replaceEditingObject` 统一剥离。
- **一个教训**：可变单例本身不是反模式；render 期变异它 + 无订阅层直读，才是（`startEditingObject` 已根治）。
- **判据**：纯 → domain；有状态/副作用 → store/services；服务端数据 → React Query；URL 能承载的 → URL。
